# Convention: Keystone Agent Context Architecture (tool.keystone-agent-context)

Defines the progressive-disclosure layering for agent instruction files in a keystone
repo. Applies to anyone adding or editing `AGENTS.md` files or conventions.

## Purpose

Agent context is expensive. Every token loaded unconditionally is a token that could
have been used for reasoning. This convention defines what belongs at each layer so
that agents receive only what is relevant to the task at hand.

## The Four Layers

1. **Global** (`~/.claude/CLAUDE.md`, generated by Nix from `archetypes.yaml`):
   - Host-level orientation and archetype label
   - Inlined: critical daily-use rules (e.g., `process.version-control`)
   - Referenced: all other conventions (loaded on-demand)
   - **Budget**: ≤ 16KB. Enforced by Nix `maxGlobalBytes` warning.
   - **OS agents**: Each provisioned OS agent (`keystone.os.agents.<name>`) receives
     its own `~/.claude/CLAUDE.md` via `modules/terminal/conventions.nix`, generated
     from `archetypes.yaml` based on the agent's `archetype` option (`engineer`,
     `product`, or custom). The **same ≤ 16KB budget applies per agent**. Archetype
     selection is therefore a context budget decision: conventions in
     `inlined_conventions` occupy this budget unconditionally; conventions in
     `referenced_conventions` are loaded on-demand only when the agent needs them.
     Choose the minimal set of inlined conventions needed for the agent's daily tasks.

2. **Project root** (`AGENTS.md` in the repo root):
   - One-paragraph project orientation
   - Navigation index → module AGENTS.md files and docs/
   - Slim module file tree (paths only, no inline descriptions)
   - Flake exports and packages tables
   - Cross-cutting gotchas that apply to ALL work in the repo
   - **Budget**: ≤ 5KB. Enforced by `agents_md_budget` .deepreview rule.

3. **Module-level** (`modules/*/AGENTS.md`, `packages/*/AGENTS.md`):
   - "How to edit THIS module" — critical options, patterns, gotchas
   - Loaded automatically when agent's CWD is inside that directory
   - Reference links to docs/ for full user-facing reference
   - **Budget**: ≤ 8KB per file (soft guideline, no automated check)

4. **On-demand references** (conventions/ and docs/):
   - `conventions/*.md`: reusable rules for any keystone project (RFC 2119 format)
   - `docs/*.md`: user-facing reference documentation (source of truth for humans)
   - Loaded only when agent explicitly reads them
   - No size budget — these are reference material

## Content Routing Rules

1. Content MUST live at the lowest layer where it is still always relevant.
   A rule that only applies when editing `modules/os/` MUST NOT be in root `AGENTS.md`.

2. Content in `conventions/` MUST be reusable across any keystone deployment.
   Keystone-specific implementation detail MUST NOT be in a convention.

3. `docs/*.md` is the authoritative source of truth for user-facing feature documentation.
   Module AGENTS.md files MUST reference docs/ rather than duplicate its content.

4. Root `AGENTS.md` MUST NOT contain option tables, CLI command listings, or prose
   documentation sections longer than 5 lines. These belong in module AGENTS.md files.

5. Content already in a convention MUST NOT be repeated in AGENTS.md.
   Add a cross-reference instead.

## Adding New Module AGENTS.md Files

6. When creating a new `modules/*/AGENTS.md`, it MUST follow the pattern:
   - H1: `# {Module} Module — Editing Guide (modules/{path}/)`
   - Sections covering: key options, gotchas, patterns unique to this module
   - Reference link to relevant `docs/*.md` at the top or bottom
   - No duplication of content already in conventions/ or root AGENTS.md

7. After creating a new module AGENTS.md, update the Module Navigation table in
   root `AGENTS.md` to include a link to the new file.

## Quality Gates

The `.deepreview` file enforces this convention automatically:
- `agents_md_budget`: fails if root `AGENTS.md` exceeds 5KB or contains inline docs
- `agents_md_dry`: flags content duplicated between AGENTS.md, conventions/, and docs/
- `conventions_review`: ensures conventions follow RFC 2119 format and cross-reference

See `.deepreview` for the rule implementations.
