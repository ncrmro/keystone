# REQ-030: Resolved Agent Identity Model

Implements: ncrmro/keystone#344

## Summary

Keystone defines one resolved agent identity model for each named agent. Every
supported CLI coding tool resolves the same identity inputs and overrides when
selecting that agent.

## Resolved Agent Model

Each named agent produces a resolved identity composed from deterministic inputs
at NixOS evaluation time, plus documented runtime inputs that load at session
start.

### Build-Time Inputs (NixOS Evaluation)

Resolved deterministically during `nixos-rebuild` / `ks build`:

| Input                | Source                                        | Description                                         |
| -------------------- | --------------------------------------------- | --------------------------------------------------- |
| `name`               | `keystone.os.agents.<name>` attrset key       | Agent identifier                                    |
| `fullName`           | `keystone.os.agents.<name>.fullName`          | Display name                                        |
| `email`              | `keystone.os.agents.<name>.email`             | Email address                                       |
| `archetype`          | `keystone.os.agents.<name>.archetype`         | Convention archetype (`engineer`, `product`)         |
| `host`               | `keystone.os.agents.<name>.host`              | Designated host                                     |
| `capabilities`       | Resolved from archetype + explicit overrides  | AI workflow capabilities                            |
| `inlinedConventions` | `archetypes.yaml` → archetype                | Convention names inlined into instruction files      |
| `referencedConventions` | `archetypes.yaml` → archetype             | Convention names available as on-demand links        |
| `notesPath`          | `keystone.os.agents.<name>.notes.path`        | Agent notes directory                               |
| `identityInputPaths` | Derived from `notesPath`                      | Paths to `SOUL.md`, `TEAM.md`, `SERVICES.md`        |
| `toolSurfaces`       | Generated by `conventions.nix`                | Paths to tool-specific instruction files             |

### Runtime Inputs

Loaded at session start by `agentctl` or the tool's native loader:

| Input                | Source                                        | Description                                         |
| -------------------- | --------------------------------------------- | --------------------------------------------------- |
| `SOUL.md`            | `{notesPath}/SOUL.md`                         | Agent identity, personality, hard constraints        |
| `TEAM.md`            | `{notesPath}/TEAM.md`                         | Team roster (humans and agents)                     |
| `SERVICES.md`        | `{notesPath}/SERVICES.md`                     | Intranet services table                             |
| Launcher preferences | `pz launcher-state-json`                      | Interactive model/provider overrides                |
| Project AGENTS.md    | `{projectDir}/AGENTS.md`                      | Project-specific context                            |

### Precedence Order

Highest priority wins. Within each layer the more specific input wins:

1. **Explicit CLI flags** — `--model`, `--role`, `--project` override everything
2. **Launcher state overrides** — project-level, then agent-level preferences
3. **NixOS agent config** — `keystone.os.agents.<name>.*` options
4. **Archetype defaults** — `archetypes.yaml` convention and role lists
5. **System-wide conventions** — `conventions.nix` generated instruction files

When no override is present, the resolved model falls back to archetype defaults
rather than leaving fields unset.

## Introspection

**REQ-030.1** `agentctl <agent> identity` MUST print the resolved identity as
JSON, showing all build-time inputs and the paths where runtime inputs are
expected.

**REQ-030.2** The resolved identity MUST be available as a NixOS read-only
option at `keystone.os.agents.<name>.resolvedIdentity` for use by other
modules and validation tests.

## Cross-Tool Parity

**REQ-030.3** Selecting the same agent name in `agentctl <name> claude`,
`agentctl <name> gemini`, `agentctl <name> codex`, and
`agentctl <name> opencode` MUST resolve the same build-time identity inputs.

**REQ-030.4** Tool-specific instruction files (`~/.claude/CLAUDE.md`,
`~/.gemini/GEMINI.md`, `~/.codex/AGENTS.md`) are generated from the same
archetype and conventions source. Format-specific adaptation is permitted;
semantic divergence is not.

## Validation

**REQ-030.5** A flake check MUST verify that the resolved identity for each
configured agent includes the expected archetype, convention lists, and
identity input paths.

**REQ-030.6** Validation MUST cover at least one archetype-default case
(`engineer`), one different archetype (`product`), and one explicit
capability override case.

## Notes-Backed Identity Inputs

**REQ-030.7** `SOUL.md`, `TEAM.md`, and `SERVICES.md` remain source inputs in
the agent's notes directory. The resolved model declares their expected paths
so that all tools load them from the same location.

**REQ-030.8** If an identity input file is missing at session start, the tool
MUST log a warning but continue with reduced context rather than failing.
