# Scaffold Agent-Space

## Objective

Create the complete agent-space directory structure including git repo initialization,
git submodule for the shared agents library, symlinks, and all operational files.

## Task

Using the SOUL.md and manifest from prior steps, build out the full agent-space repo
with all files needed for the autonomous task loop. All files go at the repo root —
the agent-space repo root IS the agent's working directory.

### Process

1. **Create directory structure**

   Under `.repos/{owner}/{repo}/`:

   ```
   .agents/                   # Will be git submodule (ncrmro/agents)
   manifests/                 # Already has modes.yaml from prior step
   SOUL.md                    # Already exists from prior step
   ```

2. **Initialize git repo (if needed)**

   Check whether `.repos/{owner}/{repo}/.git` already exists. If it does, skip
   initialization — the user may have already created the repo with a remote
   configured (e.g., Forgejo).

   ```bash
   cd .repos/{owner}/{repo}
   # Only if .git does NOT exist:
   git init
   ```

3. **Add shared agents as git submodule**

   ```bash
   cd .repos/{owner}/{repo}
   git submodule add ../../ncrmro/agents .agents
   ```

   The relative path `../../ncrmro/agents` resolves from `.repos/{owner}/{repo}/`
   to `.repos/ncrmro/agents/` — the already-cloned shared library. This ensures
   the submodule works regardless of where obsidian is checked out.

   If the submodule command fails because the path isn't a bare git URL, use the
   full Forgejo SSH URL instead:

   ```bash
   git submodule add ssh://forgejo@git.ncrmro.com:2222/ncrmro/agents.git .agents
   ```

4. **Create .deepwork symlink**

   The `.deepwork` directory is committed to the ncrmro/agents repo, so the git
   submodule clone will contain it. The symlink is at the repo root:

   ```bash
   cd .repos/{owner}/{repo}
   ln -s .agents/.deepwork .deepwork
   ```

   No `../` needed — both `.agents/` and `.deepwork` are at the repo root.

   Verify the symlink resolves: `ls .deepwork/job.schema.json` should succeed.

5. **Create TEAM.md and conventions symlinks**

   Symlink `TEAM.md` and the `conventions/` folder from the `.agents` submodule
   to the repo root. This allows for shorter, cleaner paths in `AGENTS.md`.

   ```bash
   cd .repos/{owner}/{repo}
   ln -s .agents/TEAM.md TEAM.md
   ln -s .agents/conventions conventions
   ```

6. **Copy Nix dev shell**

   The shared agents repo provides a `flake.nix` with the dev shell (deepwork, yq,
   etc.). **Copy** (not symlink) both `flake.nix` and `flake.lock` — Nix flakes
   require all paths to be tracked in the repo's git tree and cannot follow
   symlinks into submodules. Also create `.envrc` for direnv integration.

   ```bash
   cd .repos/{owner}/{repo}
   cp .agents/flake.nix flake.nix
   cp .agents/flake.lock flake.lock
   echo "use flake" > .envrc
   direnv allow
   ```

   Running `direnv allow` authorizes the `.envrc` so the dev shell activates
   automatically when entering the directory — no manual step needed.

   **Note:** When the shared agents repo updates its flake, agent-spaces need
   their copies updated too. Run from the agent-space root:

   ```bash
   cp .agents/flake.nix flake.nix
   cp .agents/flake.lock flake.lock
   ```

7. **Generate AGENTS.md**

   `AGENTS.md` is generated by `.agents/compose.sh` using the `archetype` defined
   in the manifest. This allows for role-specific core tool selections (e.g.,
   engineering vs. product) while keeping common includes dynamic.

   ```bash
   cd .repos/{owner}/{repo}
   .agents/compose.sh manifests/modes.yaml --agent-md > AGENTS.md
   ```

   The generated file uses `@` includes (`@SOUL.md`, `@HUMAN.md`,
   `@SERVICES.md`, etc.) which Claude Code resolves relative to the repo root.

8. **Create CLAUDE.md symlink**

   ```bash
   cd .repos/{owner}/{repo}
   ln -s AGENTS.md CLAUDE.md
   ```

   Symlink: `CLAUDE.md → AGENTS.md` (where AGENTS.md is a regular file)

9. **Generate PROJECTS.yaml**

   Write `PROJECTS.yaml` with the project priority list. Pull from the
   obsidian `projects/` directory to pre-populate. Include the `sources` section
   with declarative commands for email and GitHub activity fetching.

   ```yaml
   projects:
     - name: "{project_name}"
       description: "{from obsidian project README frontmatter}"
       status: active
       priority: 1
       updated: "{today}"

   sources:
     - name: "email"
       command: "himalaya envelope list --account {agent_name} -o json"
   ```

10. **Generate TASKS.yaml**

    Write `TASKS.yaml` with an empty task queue:

    ```yaml
    tasks: []
    ```

11. **Generate SCHEDULES.yaml**

Write `SCHEDULES.yaml` with sensible defaults based on the agent's role:

```yaml
schedules:
  - name: "daily-priorities"
    description: "Send daily status digest email with priorities and blockers"
    schedule: "daily"
    workflow: "daily_status/send"
```

11. **Generate ISSUES.yaml**

    Write `ISSUES.yaml` with an empty issues list:

    ```yaml
    issues: []
    ```

12. **Generate supporting files**

    Create minimal versions of:
    - `REQUIREMENTS.md` — Link to the shared agents REQUIREMENTS.md or copy
      the task loop requirements template
    - `ARCHITECTURE.md` — Task loop architecture diagram (use Drago's as template)
    - `SERVICES.md` — List available services (Forgejo, email, etc.)
    - `HUMAN.md` — Human operator contact info

13. **Create .mcp.json**

    Write `.repos/{owner}/{repo}/.mcp.json` with the MCP servers every agent needs:

    ```json
    {
      "mcpServers": {
        "deepwork": {
          "command": "deepwork",
          "args": [
            "serve",
            "--path",
            ".",
            "--external-runner",
            "claude",
            "--platform",
            "claude"
          ]
        },
        "chrome-devtools": {
          "command": "npx",
          "args": [
            "-y",
            "chrome-devtools-mcp@latest",
            "--browserUrl",
            "http://localhost:9222"
          ]
        }
      }
    }
    ```

    See convention `tool.chrome-devtools` for Chrome DevTools requirements.

14. **Create .gitignore**

    Write `.repos/{owner}/{repo}/.gitignore`:

    ```
    .direnv/
    result
    ```

15. **Verify structure**

    Run `find .repos/{owner}/{repo} -type f | sort` and confirm all expected files exist.
    Verify symlinks resolve correctly with `ls -la`.

## Output Format

Report all created files as the `agent_space_files` output. The complete file tree
should look like:

```
.repos/{owner}/{repo}/
  .gitignore
  .envrc
  .mcp.json
  .agents/                         # git submodule (ncrmro/agents)
  .deepwork -> .agents/.deepwork
  flake.nix                        # copied from .agents/
  flake.lock                       # copied from .agents/
  SOUL.md
  AGENTS.md -> .agents/AGENTS_TEMPLATE.md
  CLAUDE.md -> AGENTS.md
  PROJECTS.yaml
  TASKS.yaml
  SCHEDULES.yaml
  ISSUES.yaml
  REQUIREMENTS.md
  ARCHITECTURE.md
  SERVICES.md
  HUMAN.md
  manifests/
    modes.yaml
```

## Quality Criteria

- All operational files (SOUL.md, AGENTS.md, etc.) are at the repo root, NOT in a notes/ subdirectory
- A .agents git submodule exists and points to the shared agents repo (ncrmro/agents)
- .deepwork is a relative symlink at the repo root pointing to .agents/.deepwork
- .mcp.json contains both `deepwork` and `chrome-devtools` MCP server entries
- AGENTS.md is generated by .agents/compose.sh (NOT a symlink)
- CLAUDE.md is a symlink to AGENTS.md
- PROJECTS.yaml, TASKS.yaml, SCHEDULES.yaml, and ISSUES.yaml contain valid YAML with correct top-level keys
- All symlinks resolve correctly (test with `ls -la`)
- Git repo exists (either pre-existing or freshly initialized)

## Context

This step does the heavy lifting of creating the actual files. The directory layout
mirrors Drago's proven structure at `.repos/drago/agent-space/`. The repo root IS the
agent's working directory — there is no `notes/` subdirectory. The git submodule
approach means each agent-space pins a specific version of the shared library and can
update independently. The .deepwork symlink gives the agent access to shared DeepWork
jobs without duplicating them.
